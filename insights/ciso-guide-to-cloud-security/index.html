The CISO's Guide to Cloud Security

Where most cloud security programs fall short, and what to get right before your next major cloud initiative.

March 31, 2025

Cloud Security in 2025: Where Programs Are Falling Short

The move to cloud has been largely complete for several years. What hasn’t kept pace is cloud security maturity. Most organizations have workloads running in AWS, Azure, or GCP — and most have security programs that were designed for on-premise environments and retrofitted, imperfectly, to the cloud.

This creates predictable gaps. Here’s where we see programs fall short most consistently, and what to do about it.

Identity Is the New Perimeter — And Most Programs Treat It Like an Afterthought

In a cloud environment, identity is the primary control boundary. Overpermissioned roles, unused credentials, and misconfigured federation are the most common initial access vectors in cloud incidents — more common than unpatched software or network intrusion.

Getting identity right means:

  • Least-privilege by default — roles granted only what’s required, reviewed regularly
  • Credential hygiene — no long-lived access keys, MFA enforced on all human identities, service accounts scoped tightly
  • Federation audit — understanding how your identity provider connects to cloud environments and what access it grants

AWS IAM, Azure AD, and GCP IAM are all powerful tools. They’re also consistently misconfigured. An identity assessment is often the highest-value first step for organizations that have been in the cloud for a while without a structured review.

Visibility: You’re Flying Blind Without It

Cloud environments generate enormous volumes of log data across compute, storage, network, and identity layers. Most organizations are capturing some of it. Very few are using it effectively.

Effective cloud visibility means:

  • CloudTrail / Audit Log ingestion and retention
  • Alerting on high-signal events — console login without MFA, unusual API calls, new admin role assignments
  • GuardDuty or equivalent threat detection enabled and tuned
  • Regular review of findings (not just “enabled”)

The most common failure mode: logging is enabled, data is being sent somewhere, but no one is looking at it and there’s no alert path for high-severity findings.

Shared Responsibility — And Where Organizations Get It Wrong

Cloud providers are responsible for the security of the cloud. You are responsible for security in the cloud. This division is well-documented but consistently misunderstood in practice.

The gaps that result:

  • Data classification — S3 buckets and blob storage with overly permissive access, sometimes public
  • Encryption — at-rest and in-transit encryption not enforced, customer-managed keys not used where required by policy
  • Patching — operating systems and applications running in cloud VMs are your responsibility; cloud-native services are the provider’s

Before Your Next Cloud Initiative

The right time to address cloud security is before a new major workload goes in, not after. A pre-launch security architecture review — scoped to the specific initiative — can identify control gaps while they’re inexpensive to fix.

The questions worth answering before you go live:

  1. What data will this workload process, and what’s the classification?
  2. What identity access model is in place, and who has admin rights?
  3. What logging and alerting will be active at launch?
  4. What’s the incident response plan specific to this environment?
  5. Who owns ongoing security operations for this workload?
